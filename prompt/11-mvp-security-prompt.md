Мне нужно добавить блок «Безопасность MVP».

Цель:
Описать и реализовать базовые меры безопасности MVP, чтобы данные пользователей были защищены не только frontend-логикой, но и ограничениями на уровне базы данных.

Основные риски, которые нужно закрыть:
- доступ к чужим данным;
- изменение пользовательских записей в обход логики приложения;
- хранение чувствительных ключей в клиенте;
- доверие только frontend без ограничений на уровне базы.

Действуй шаг за шагом.

1. Проанализируй основные риски MVP
Опиши следующие риски простыми словами:

Риск 1: доступ к чужим данным
Поясни, что пользователь может попытаться получить продукты, покупки, профиль или подписку другого пользователя через прямой запрос к Supabase.

Риск 2: изменение пользовательских записей в обход логики приложения
Поясни, что даже если интерфейс не показывает кнопку редактирования чужих данных, пользователь может открыть DevTools или отправить запрос напрямую.

Риск 3: хранение чувствительных ключей в клиенте
Поясни, что в frontend нельзя хранить:
- Supabase service role key;
- Stripe secret key;
- приватные API tokens;
- админские ключи.

В клиенте можно использовать только publishable/anon key при условии, что включён RLS.

Риск 4: доверие только frontend
Поясни, что frontend можно изменить или обойти, поэтому настоящие ограничения должны быть на уровне базы данных.

2. Создай миграцию безопасности
Создай файл:
projects/supabase/007_security_hardening.sql

В нём нужно включить и настроить Row Level Security для таблиц:
- profiles;
- user_settings;
- products;
- shopping_items;
- product_events;
- subscriptions;
- recipes.

3. Настрой безопасность profiles
Для таблицы profiles включи RLS.

Добавь policies:
- пользователь может читать только свой профиль;
- пользователь может создать только свой профиль;
- пользователь может обновить только свой профиль.

Проверка:
auth.uid() = id

4. Настрой безопасность user_settings
Для таблицы user_settings включи RLS.

Добавь policies:
- пользователь может читать только свои настройки;
- пользователь может создать только свои настройки;
- пользователь может обновить только свои настройки.

Проверка:
auth.uid() = user_id

5. Настрой безопасность products
Для таблицы products включи RLS.

Добавь policies:
- пользователь может читать только свои продукты;
- пользователь может добавлять продукты только себе;
- пользователь может обновлять только свои продукты;
- пользователь может удалять только свои продукты.

Проверка:
auth.uid() = user_id

Это должно защищать холодильник каждого пользователя.

6. Настрой безопасность shopping_items
Для таблицы shopping_items включи RLS.

Добавь policies:
- пользователь может читать только свой список покупок;
- пользователь может добавлять товары только себе;
- пользователь может обновлять только свои товары;
- пользователь может удалять только свои товары.

Проверка:
auth.uid() = user_id

Это должно защищать список покупок каждого пользователя.

7. Настрой безопасность product_events
Для таблицы product_events включи RLS.

Добавь policies:
- пользователь может читать только свои события;
- пользователь может добавлять события только себе.

Не добавляй policies на update/delete для обычного клиента.

Поясни, что product_events — это история действий, поэтому её не стоит свободно редактировать с frontend.

8. Настрой безопасность subscriptions
Для таблицы subscriptions включи RLS.

Добавь policies:
- пользователь может читать только свою подписку;
- пользователь может создать только свою подписку;
- в MVP test mode пользователь может обновить свою подписку.

Обязательно добавь комментарий:
В production обновление subscriptions из frontend нужно убрать. Статус подписки должен обновляться только через backend или Stripe webhook.

9. Настрой безопасность recipes
Таблица recipes — публичный каталог рецептов.

Настрой её так:
- любой пользователь может читать только активные рецепты;
- frontend не может добавлять рецепты;
- frontend не может обновлять рецепты;
- frontend не может удалять рецепты.

Добавь комментарий, что изменения каталога рецептов должны выполняться только через миграции или admin role.

10. Примени миграцию в Supabase
Примени миграцию security_hardening к Supabase-проекту.

После применения проверь, что RLS policies созданы без ошибок.
