Домашнее задание 11 — Безопасность MVP
Проект: Chill — умный помощник для управления продуктами и кухней

Цель
Добавить базовый уровень безопасности MVP, чтобы данные пользователей были защищены не только логикой frontend, но и ограничениями на уровне базы данных Supabase.

Зачем нужна безопасность
В MVP уже есть регистрация, профили, продукты, список покупок, события продуктов, подписки Pro и рецепты. Это значит, что приложение работает с пользовательскими данными.

Если защищать данные только через JavaScript на сайте, пользователь может открыть DevTools, изменить запросы или напрямую обратиться к Supabase API. Поэтому основные ограничения должны быть не только во frontend, но и на уровне базы данных.

Основные риски

Риск 1: доступ к чужим данным
Проблема:
Если в базе нет ограничений, один пользователь может попытаться прочитать продукты, покупки, профиль или подписку другого пользователя.

Пример риска:
Пользователь A может изменить запрос и попробовать получить записи пользователя B.

Решение:
Для пользовательских таблиц включён Row Level Security. Доступ разрешён только к строкам, где auth.uid() совпадает с user_id или id пользователя.

Риск 2: изменение пользовательских записей в обход логики приложения
Проблема:
Даже если интерфейс не показывает кнопку редактирования чужих данных, пользователь может отправить прямой запрос к Supabase и попытаться изменить чужую запись.

Решение:
Для таблиц products, shopping_items, user_settings, profiles и subscriptions добавлены RLS policies на insert, update, delete. Каждая операция проверяет владельца записи.

Риск 3: хранение чувствительных ключей в клиенте
Проблема:
В frontend нельзя хранить секретные ключи, например Stripe Secret Key или Supabase Service Role Key. Всё, что лежит в браузере, может увидеть пользователь.

Решение:
В клиенте используется только publishable/anon key Supabase. Такой ключ безопасен только при условии, что включён RLS. Секретные ключи должны храниться только на backend или в Supabase Edge Functions.

Риск 4: доверие только frontend без ограничений на уровне базы
Проблема:
Frontend можно обойти. Нельзя считать, что пользователь будет нажимать только разрешённые кнопки.

Решение:
Все важные ограничения перенесены в Supabase через RLS policies. Даже если пользователь отправит запрос вручную, база проверит права доступа.

Безопасность profiles
Для таблицы profiles включён RLS.

Правила:
- пользователь может читать только свой профиль;
- пользователь может создать только свой профиль;
- пользователь может обновить только свой профиль.

Проверка:
auth.uid() = id

Это защищает email, имя и профиль пользователя от просмотра другими аккаунтами.

Безопасность user_settings
Для таблицы user_settings включён RLS.

Правила:
- пользователь может читать только свои настройки;
- пользователь может создать только свои настройки;
- пользователь может обновить только свои настройки.

Проверка:
auth.uid() = user_id

Это нужно, чтобы настройки одного пользователя не могли быть изменены другим пользователем.

Безопасность products
Для таблицы products включён RLS.

Правила:
- пользователь может читать только свои продукты;
- пользователь может добавлять продукты только себе;
- пользователь может обновлять только свои продукты;
- пользователь может удалять только свои продукты.

Проверка:
auth.uid() = user_id

Это защищает холодильник каждого пользователя.

Безопасность shopping_items
Для таблицы shopping_items включён RLS.

Правила:
- пользователь может читать только свой список покупок;
- пользователь может добавлять товары только в свой список;
- пользователь может отмечать купленными только свои товары;
- пользователь может удалять только свои товары.

Проверка:
auth.uid() = user_id

Это защищает список покупок каждого пользователя.

Безопасность product_events
Для таблицы product_events включён RLS.

Правила:
- пользователь может читать только свои события;
- пользователь может добавлять события только себе.

Изменение и удаление событий из клиента не разрешено.

Почему так:
product_events — это история действий. Например, когда пользователь нажал «Съел», создаётся событие. Исторические записи лучше не давать свободно редактировать с frontend, чтобы аналитика оставалась честнее.

Безопасность subscriptions
Для таблицы subscriptions включён RLS.

Правила:
- пользователь может читать только свою подписку;
- пользователь может создать только свою подписку;
- в MVP-режиме пользователь может обновить свою подписку.

Важно:
Обновление подписки с frontend разрешено только для учебного MVP, потому что сейчас есть тестовая кнопка «Активировать Premium в тестовом режиме».

В production это правило нужно убрать. Статус подписки должен обновляться только через backend или Stripe webhook.

Безопасность recipes
Таблица recipes является публичным каталогом рецептов.

Правила:
- любой пользователь может читать только активные рецепты;
- frontend не может добавлять, менять или удалять рецепты.

Почему так:
Рецепты — это общий справочник. Пользователи могут его смотреть, но не должны менять общий каталог через клиент.

Что такое RLS
RLS — Row Level Security. Это механизм Supabase/PostgreSQL, который ограничивает доступ не просто к таблице целиком, а к отдельным строкам.

Пример:
Даже если пользователь отправит запрос:

select * from products;

База вернёт только те строки, где:

auth.uid() = user_id

То есть пользователь увидит только свои продукты.

Почему нельзя полагаться только на frontend
Frontend работает в браузере пользователя. Значит пользователь может:
- открыть DevTools;
- изменить JavaScript;
- отправить запрос напрямую к Supabase;
- подменить id записи;
- попробовать получить чужие данные.

Поэтому frontend — это только удобный интерфейс. Настоящая защита должна быть на уровне базы данных.

Что осталось важно помнить
В клиенте можно хранить:
- Supabase anon key;
- Supabase publishable key;
- публичные настройки.

В клиенте нельзя хранить:
- Supabase service role key;
- Stripe secret key;
- ключи админ-доступа;
- приватные API-токены.

Для настоящих платежей нужен backend
Сейчас в MVP есть тестовая активация Pro. Это нормально для демонстрации.

Но реальная схема должна быть такой:

Пользователь нажимает «Купить Pro»
↓
Frontend отправляет запрос на backend
↓
Backend создаёт Stripe Checkout Session
↓
Пользователь оплачивает
↓
Stripe отправляет webhook на backend
↓
Backend обновляет subscriptions
↓
Frontend видит status = active

Такой подход безопаснее, потому что пользователь не может сам поставить себе статус active.

Итоговая схема безопасности

Пользователь входит через Supabase Auth
↓
Supabase выдаёт auth.uid()
↓
Frontend отправляет запрос в базу
↓
RLS проверяет auth.uid()
↓
База возвращает только разрешённые строки
↓
Пользователь видит только свои данные

Выводы
- Главный риск MVP — доступ к чужим данным.
- Frontend нельзя считать защитой, потому что его можно обойти.
- RLS защищает данные на уровне базы.
- Пользователь может работать только со своими продуктами, покупками, настройками и подпиской.
- Рецепты сделаны публичными, но read-only.
- Секретные ключи нельзя хранить в браузере.
- Для реальной оплаты нужен backend и webhook.
