Домашнее задание 7 — Аутентификация пользователей
Проект: Chill — умный помощник для управления продуктами и кухней

Что сделано
В проект Chill была добавлена аутентификация пользователей через Supabase Auth. Теперь пользователь может зарегистрироваться, войти в аккаунт, выйти из аккаунта и работать со своим личным профилем.

Также была исправлена проблема, при которой продукты и покупки были общими для всех пользователей. Теперь у каждого аккаунта отдельный набор данных.

Улучшение 1: Добавление регистрации и входа
Что:
На сайт был добавлен блок аутентификации:
- поле Email
- поле Пароль
- кнопка Войти
- кнопка Регистрация
- кнопка Выйти после входа

Зачем:
Без аккаунтов приложение могло хранить данные только локально в браузере. После добавления аутентификации пользователь может войти в свой аккаунт и хранить данные в Supabase.

Как реализовано:
- подключён Supabase Auth
- добавлен Supabase CDN в index.html
- создан файл projects/supabase/supabase.js
- создан файл projects/supabase/app-sync.js
- добавлены функции signUp, signIn и signOut

Улучшение 2: Добавление профиля пользователя
Что:
После входа пользователь видит карточку профиля.

В профиле отображается:
- email пользователя
- имя пользователя, если оно есть
- количество продуктов
- количество покупок
- статус облачной синхронизации

Зачем:
Профиль показывает, что пользователь действительно вошёл в аккаунт, а данные теперь синхронизируются через облако.

Как реализовано:
- создан файл projects/supabase/profile-ui.js
- добавлен метод getProfile()
- добавлен метод updateProfile()
- профиль связан с таблицей profiles
- карточка профиля создаётся динамически через JavaScript

Улучшение 3: Разделение данных по пользователям
Что:
Была исправлена проблема, когда разные пользователи видели одинаковые продукты и покупки.

Теперь каждый пользователь видит только свои данные:
- свои продукты
- свой список покупок
- свои события
- свои избранные рецепты
- свои настройки

Зачем:
Для приложения с аккаунтами важно, чтобы данные одного пользователя не были доступны другому. Это базовая безопасность и правильная логика работы сервиса.

Как реализовано:
- в таблицах используется поле user_id
- включён Row Level Security в Supabase
- добавлены RLS policies
- запросы getProducts() и getShoppingItems() фильтруются по текущему user_id
- deleteProduct() удаляет только записи текущего пользователя

Улучшение 4: Исправление подтверждения email
Что:
Изначально Supabase требовал подтверждение email, из-за чего пользователь не мог войти сразу после регистрации.

Что было сделано:
- подтверждение email было отключено в Supabase Dashboard
- уже созданные неподтверждённые аккаунты были подтверждены вручную
- текст ошибки на сайте был улучшен

Зачем:
Так регистрация стала проще: пользователь может создать аккаунт и сразу войти без перехода в почту.

Процесс работы
Шаг 1: Подключение Supabase Auth
Были добавлены функции регистрации, входа и выхода через Supabase.

Шаг 2: Добавление UI авторизации
Форма входа и регистрации была добавлена на сайт и затем перенесена в правый верхний угол, чтобы не портить внешний вид главной страницы.

Шаг 3: Добавление профиля
Была создана отдельная карточка профиля пользователя с email и статистикой.

Шаг 4: Исправление ошибок регистрации
Были улучшены сообщения об ошибках при регистрации и входе.

Шаг 5: Отключение подтверждения email
Подтверждение email было отключено, чтобы пользователи могли входить сразу после регистрации.

Шаг 6: Изоляция пользовательских данных
В Supabase был включён RLS, добавлены политики безопасности, а frontend-запросы были дополнительно отфильтрованы по текущему пользователю.

Выводы
- Supabase Auth позволяет быстро добавить регистрацию и вход
- Профиль пользователя делает приложение более похожим на полноценный сервис
- RLS обязателен, если у каждого пользователя должны быть свои данные
- Отключение подтверждения email упрощает вход для MVP
- После добавления auth проект стал ближе к настоящему SaaS-приложению
