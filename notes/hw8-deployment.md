Домашнее задание 8 — Публикация MVP
Проект: Chill — умный помощник для управления продуктами и кухней

Цель
Опубликовать MVP так, чтобы на него можно было зайти по постоянной ссылке, и понять, какие варианты деплоя существуют и почему выбран именно этот.

Что сделано
MVP проекта Chill был опубликован в интернете через Render. Теперь приложение доступно по постоянной ссылке:

https://chill-ggoj.onrender.com/

Это значит, что проект больше не нужно запускать только локально через компьютер разработчика. Пользователь может открыть сайт в браузере с любого устройства.

Почему нужен деплой
До деплоя проект можно было открыть только локально:
- через файлы index.html, style.css, script.js
- через локальный сервер Node.js
- через адрес http://localhost:3000

Но localhost работает только на компьютере разработчика. Для реального MVP нужна публичная ссылка, которую можно отправить преподавателю, пользователю или команде.

Варианты деплоя

Вариант 1: GitHub Pages
GitHub Pages подходит для статических сайтов.

Плюсы:
- бесплатно
- просто настроить
- хорошо подходит для HTML/CSS/JS
- можно быстро показать лендинг или простой сайт

Минусы:
- нет backend-сервера
- нельзя полноценно запускать Express
- ограниченные возможности для API
- не подходит для полноценного full-stack приложения

Для Chill этот вариант был бы удобен только на раннем этапе, когда проект был обычным статическим сайтом.

Вариант 2: Vercel
Vercel хорошо подходит для frontend-приложений и современных фреймворков вроде React, Next.js, Vue.

Плюсы:
- быстрый деплой
- удобная интеграция с GitHub
- автоматическое обновление после commit
- хорошо подходит для frontend

Минусы:
- Express-приложение нужно адаптировать под serverless
- проект Chill уже использует обычный Node.js сервер
- для текущей структуры Render проще

Vercel можно использовать в будущем, если проект будет переведён на React или Next.js.

Вариант 3: Railway
Railway подходит для backend и full-stack приложений.

Плюсы:
- поддерживает Node.js
- можно запускать backend
- удобно подключать базы данных
- есть интеграция с GitHub

Минусы:
- бесплатные лимиты могут меняться
- интерфейс и настройки могут быть сложнее для новичка
- для учебного MVP Render оказался проще

Railway тоже подходит для Chill, но в данном случае был выбран Render.

Вариант 4: Render
Render — платформа для публикации backend и full-stack приложений.

Плюсы:
- поддерживает Node.js и Express
- легко подключается к GitHub
- автоматически деплоит проект после commit
- можно указать Build Command и Start Command
- подходит для full-stack MVP
- даёт публичную ссылку

Минусы:
- на бесплатном тарифе сервер может засыпать
- первый запуск после паузы может быть медленным

Почему выбран Render
Для Chill был выбран Render, потому что проект уже был переведён в full-stack формат:
- frontend находится в папке projects
- backend находится в папке backend
- сервер запускается через Express
- в корне проекта есть package.json
- приложение запускается командой node backend/server.js

Render хорошо подходит под такую архитектуру, потому что может запустить обычный Node.js сервер без серьёзной переработки проекта.

https://chill-ggoj.onrender.com/

Что получилось в итоге
После деплоя приложение Chill стало доступно онлайн.

Теперь пользователь может:
- открыть сайт по ссылке
- зарегистрироваться
- войти в аккаунт
- добавить продукты
- добавить покупки
- использовать данные через Supabase
- работать не только локально, но и через интернет

Как работает обновление сайта
Render подключён к GitHub. Это значит:

1. Изменения вносятся в репозиторий.
2. Создаётся commit.
3. Render видит изменение в ветке main.
4. Render автоматически запускает новый deploy.
5. Через некоторое время новая версия появляется на сайте.

Это удобно, потому что не нужно каждый раз вручную загружать файлы на сервер.

Выводы
- Для простого статического сайта можно использовать GitHub Pages.
- Для frontend-проектов на React/Next.js хорошо подходит Vercel.
- Для full-stack Node.js приложений удобно использовать Render или Railway.
- Для Chill выбран Render, потому что проект использует Express backend.
- Деплой дал проекту постоянную публичную ссылку.
- Автодеплой через GitHub упростил дальнейшую разработку.

Что можно улучшить дальше
- подключить собственный домен
- настроить production environment variables
- добавить страницу статуса сервера
- настроить логирование ошибок
- добавить CI/CD проверки
- подключить мониторинг
- оптимизировать cold start на бесплатном тарифе
