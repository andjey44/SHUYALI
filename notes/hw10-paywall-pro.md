Домашнее задание 10 — Paywall и аккаунты разных типов
Проект: Chill — умный помощник для управления продуктами и кухней

Цель
Добавить первую монетизацию MVP: сделать разные типы аккаунтов, пробный период на 7 дней и экран подписки Pro, который нужен для продолжения работы сайта после окончания trial.

Что было сделано
В проект Chill была добавлена MVP-реализация подписки Pro.

Теперь у пользователя может быть один из нескольких статусов:
- гостевой режим;
- пробный период;
- Pro активен;
- пробный период закончился;
- подписка отменена или просрочена.

Для учебного MVP реальная оплата пока не подключена. Вместо неё сделана тестовая активация Pro, чтобы показать механику монетизации без работы с настоящими платежами.

Зачем нужна подписка
До этого сайт был полностью бесплатным. Но для настоящего продукта важно заранее продумать модель монетизации.

Для Chill логичная модель:
1. Пользователь регистрируется.
2. Получает 7 дней бесплатного доступа.
3. После окончания пробного периода сайт показывает paywall.
4. Для продолжения работы нужна подписка Pro.
5. В будущем тестовая кнопка заменяется на настоящую оплату через Stripe или другой платёжный сервис.

Что входит в Pro
В Pro были вынесены основные функции сайта:
- облачное хранение продуктов;
- список покупок;
- рецепты из остатков;
- аналитика кухни;
- синхронизация между устройствами.

Гостевой режим при этом не ломается: если пользователь не вошёл в аккаунт, сайт всё ещё может работать локально через localStorage.

Новая таблица subscriptions
В Supabase была добавлена таблица:

subscriptions

Она хранит информацию о подписке пользователя.

Основные поля:
- id — уникальный ID подписки;
- user_id — пользователь, которому принадлежит подписка;
- plan — тариф, например premium;
- status — статус подписки;
- trial_started_at — дата начала пробного периода;
- trial_ends_at — дата окончания пробного периода;
- current_period_end — конец оплаченного периода;
- stripe_customer_id — ID клиента Stripe на будущее;
- stripe_subscription_id — ID подписки Stripe на будущее;
- created_at — дата создания;
- updated_at — дата обновления.

Статусы подписки
Для MVP были предусмотрены такие статусы:

trialing
Пользователь находится в пробном периоде. Сайт работает бесплатно 7 дней.

active
Pro активен. Все функции сайта доступны.

past_due
Оплата просрочена. Этот статус пригодится при подключении реальных платежей.

canceled
Подписка отменена.

expired
Пробный период закончился, доступа к Pro-функциям нет.

Row Level Security
Для таблицы subscriptions включён RLS.

Это нужно, чтобы пользователь мог видеть и изменять только свою подписку.

Правило:
- пользователь может читать только строку, где auth.uid() = user_id;
- пользователь может создать только свою подписку;
- в MVP-режиме пользователь может обновить свою подписку для тестовой активации Pro.

Важно: в реальном продукте обновление статуса подписки нельзя делать с frontend. Это должен делать backend или webhook от платёжной системы.

Автоматический trial на 7 дней
Для новых пользователей добавлено автоматическое создание подписки.

Логика:

Новый пользователь регистрируется
↓
Supabase создаёт профиль
↓
Supabase создаёт user_settings
↓
Supabase создаёт subscriptions
↓
Пользователь получает trial на 7 дней

Также для уже существующих пользователей была добавлена подписка, если её ещё не было.

Почему пока без реальной оплаты
Реальные платежи требуют:
- платёжного провайдера;
- backend endpoint для создания платежа;
- webhook для подтверждения оплаты;
- хранения Stripe secret key только на сервере;
- обработки отмены подписки и неуспешной оплаты.

Поэтому для MVP был выбран безопасный учебный вариант: показать архитектуру подписки и paywall без списания денег.

Как будет работать реальная оплата в будущем
В будущем тестовую активацию можно заменить на Stripe Checkout.

Схема:

Пользователь нажимает «Купить Pro»
↓
Frontend отправляет запрос на backend
↓
Backend создаёт Stripe Checkout Session
↓
Пользователь оплачивает подписку
↓
Stripe отправляет webhook
↓
Backend обновляет subscriptions в Supabase
↓
status становится active
↓
Paywall исчезает, функции сайта доступны

Почему статус нельзя менять с frontend в production
Сейчас activatePremiumTestMode() обновляет подписку напрямую из frontend, потому что это MVP-демонстрация.

В настоящем продукте так делать нельзя, потому что пользователь мог бы подменить запрос и активировать Pro без оплаты.

Правильный production-подход:
- frontend только открывает страницу оплаты;
- backend создаёт checkout session;
- Stripe подтверждает оплату через webhook;
- только backend меняет статус подписки на active.
